# 知识库元结构

## 当前领域（顶层骨架）

## 语义类型（Semantic Types）

卡片的类型由 frontmatter 的 `type` 字段表达。目录结构**不按类型组织**，而是按**领域**组织。

| type | 定义 | 回答的问题 | 例子 |
|------|------|-----------|------|
| `domain` | 领域卡片，知识空间的顶层骨架 | 「这是哪个思想领域？」 | 教学领域、认知领域 |
| `notion` | 观念、立场、信条、价值判断 | 「我认为/怎么看这件事？」 | 学习是结构生成 |
| `principle` | 原则、原理、规律、客观约束 | 「什么必须被遵守？为什么会这样？」 | 认知负荷有限原则 |
| `phenomenon` | 现象、模式、趋势、可观察事实 | 「发生了什么？呈现什么模式？」 | 学生依赖脚手架 |
| `entity` | 有明确边界的独立实体对象 | 「这个具体对象是什么？」 | Vygotsky、某本书、某所学校 |
| `group` | 由多个实体或事物构成的群体/集合 | 「这是哪一类人/事物的群体？」 | 建构主义学派 |
| `model` | 模型、框架、解释结构 | 「这个复杂事物怎么理解？」 | 双系统模型、学习螺旋 |
| `method` | 方法、技巧、流程、操作步骤 | 「怎么做才能达到目标？」 | 费曼技巧、间隔重复 |
| `conflict` | 冲突、张力、两难、对立 | 「这里的根本矛盾是什么？」 | 支持 vs 自主的张力 |
| `note` | 暂时无法归类的想法 | 「这个想法还没成熟」 | 待后续迁移 |

### 类型边界判定

当一张卡片可能属于多个类型时，按以下优先级选择**主类型**（其他维度通过 `relations` 关联）：

1. 如果它主要约束判断或行动 → `principle`
2. 如果它主要解释某个复杂事物的结构 → `model`
3. 如果它主要提供可执行步骤 → `method`
4. 如果它主要描述可观察模式或趋势 → `phenomenon`
5. 如果它主要表达个人立场或价值判断 → `notion`
6. 如果它是有明确边界的独立对象 → `entity`
7. 如果它是多个对象构成的群体 → `group`
8. 如果它记录根本性的对立或张力 → `conflict`

### 典型交叉案例

**费曼技巧**
- 主类型：`method`（因为它首要回答「怎么做」）
- 关联：`principle`「解释能暴露理解漏洞」、`model`「知识分层模型」

**认知负荷理论**
- 若强调整体理论框架 → `model`
- 若只强调「工作记忆容量有限」这一约束 → `principle`

**建构主义学派**
- 主类型：`group`（一群人/思想传统的集合）
- 关联：`entity`「Piaget、Vygotsky」、`notion`「学习是主动建构」

## 目录组织原则

- 领域卡片统一放在 `01-Cards/domains/`。
- 实例卡片按 `/construct` 或 `/digest` 决定的**领域目录**存放。
- 目录命名使用 `kebab-case`，与领域名称一致。
- 同一领域目录下可以混合存放多种语义类型的卡片。
- 子目录数量建议控制在 10 个以内，超过时考虑合并或提升为更高级的领域。

## 允许的 relation 类型

- `belongs-to`：属于某个领域或概念。
- `extends`：扩展、深化某个概念。
- `conflicts-with`：与某个概念冲突。
- `supports`：支持某个概念或立场。
- `example-of`：某个概念的具体例子。
- `applies-to`：应用于某个场景或问题。
- `influences`：影响某个概念或领域。
- `implements`：某方法实现了某原则或模型。
- `based-on`：基于某原则、模型或观念。

## 命名约定

- `id`：`kebab-case`，全局唯一。
- 文件名：与 `id` 一致，例如 `cognitive-scaffolding.md`。
- 中文标题放在 `title` 字段和 `# 标题` 中。
- 标签：`kebab-case` 或中文，小写。

## 实例引用上限

- `01-Cards/` 卡片：原始资料实例 ≤ 4 条。
- `02-Profile/` 档案：原文摘录 ≤ 10 条。

## 类型扩展规则

- 新增 type 需满足：有 ≥3 张卡片需要它，且无法用现有 type 表达。
- 新增 type 必须同步更新 `ontology.md` 中的定义、边界判定和聚合方向。
- type 数量建议控制在 10 个以内，超过时需考虑合并或提升为子领域。
