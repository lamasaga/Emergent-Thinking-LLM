# AGENTS.md — AI 操作手册（新引擎版）

> **核心原则**：AI 不是来"分解文章"的，AI 是来"比较和消化"的。

---

## 一、系统的核心逻辑

### 1.1 关系卡片是骨架

知识网络的组织方式不是文件夹，不是标签，而是**关系**。

- 顶级关系卡片（如"教学关系""认知关系"）是AI的**入口**
- 子关系卡片（如"支持关系""组织关系"）是AI的**导航**
- 概念/实体卡片是AI**滋养**的对象

### 1.2 AI 的渐进式阅读

处理新文章时，AI **不允许**一次性读取全部卡片。必须遵循：

```
Step 1: 读取 01-Cards/relations/ 下的全部顶级关系卡片
Step 2: 判断文章涉及哪些关系类型
Step 3: 读取相关的子关系卡片
Step 4: 读取子关系卡片中提到的具体概念/实体/冲突卡片
Step 5: 进行比较、补充、冲突检测
Step 6: 更新关系卡片（如有新发现）
```

### 1.3 文章是燃料，不是资产

**禁止行为**：
- ❌ 为每篇文章创建摘要文件
- ❌ 把文章拆成碎片分发到无数空卡片
- ❌ 保留大量未消化的原始文档在 Inbox

**必须行为**：
- ✅ 用文章的内容去**丰富已有卡片**
- ✅ 只在遇到**真正新的概念**时才新建卡片
- ✅ 处理完后将原始文档移动到 03-Archive/

## 二、消化指令（/digest）

**触发条件**：00-Inbox/ 中有新文件

**执行步骤**：

```
1. 读取 00-Inbox/ 中的新文件
2. 读取 01-Cards/relations/ 下的全部顶级关系卡片
3. 判断文章涉及哪些关系类型（可能涉及多个）
4. 读取相关的子关系卡片
5. 读取子关系涉及的具体概念/实体/冲突卡片
6. 进行比较分析：
   a. 文章中的概念是否已有卡片？
      → 有：丰富那张卡片（补充例子、引用、角度、修正）
      → 无：判断是否需要新建（重要且持久的概念才新建）
   b. 文章内容与现有卡片是否冲突？
      → 是：在卡片中标记冲突，考虑新建冲突卡片
   c. 文章中的实体是否已有卡片？
      → 有：补充信息
      → 无：判断重要性后决定是否新建
7. 更新关系卡片（如果发现了新的关联）
8. 将原始文档移动到 03-Archive/
9. 输出消化报告：
   - 丰富了哪些卡片
   - 新建了哪些卡片
   - 发现了哪些冲突
   - 更新了哪些关系
```

**丰富已有卡片的规则**：

当决定丰富一张已有卡片时，在卡片的对应部分追加内容：

```markdown
## 实例（新增）

- **来自《xxx》**：文章提到的具体案例...
```

**新建卡片的规则**：

只有当满足以下条件时才新建：
1. 这个概念/实体是**重要的**（不是文章中一闪而过的次要概念）
2. 这个概念/实体是**持久的**（不是只在某一篇文章中出现的临时概念）
3. 已有卡片体系中**没有**可以承载它的卡片

新建卡片必须有**实质性内容**，禁止空模板。

**冲突检测规则**：

冲突不是"不同说法"，而是**根本性的张力**：

| 不同说法 | 冲突 |
|---------|------|
| "脚手架应该尽早撤除" vs "脚手架应该逐步撤除" | ❌ 不是冲突，是程度差异 |
| "脚手架应该存在" vs "脚手架不应该存在" | ✅ 是冲突，是立场对立 |
| "元认知包括 planning" vs "元认知包括 monitoring" | ❌ 不是冲突，是互补 |
| "学习是知识累积" vs "学习是结构生成" | ✅ 是冲突，是范式对立 |

发现冲突后：
1. 在涉及的概念卡片中添加冲突标记
2. 检查 conflicts/ 下是否已有相关冲突卡片
3. 如有，丰富该冲突卡片
4. 如无，新建冲突卡片

## 三、关系卡片的维护指令（/relate）

**触发条件**：新卡片创建后，或发现新的关系模式后

**执行步骤**：

```
1. 读取新创建的卡片
2. 判断这张卡片属于哪些关系类型
3. 读取对应的关系卡片
4. 在关系卡片的对应部分添加链接
5. 检查是否需要新建子关系卡片
```

**示例**：新建了"同伴互助"概念卡片

→ 读取"支持关系"卡片
→ 在"支持关系"的"涉及的概念"部分添加：`- **同伴互助**`
→ 判断是否需要新建"同伴互助关系"作为"支持关系"的子关系

## 四、认知档案的使用（/profile）

**触发条件**：处理 00-Inbox/my-thoughts/ 下的文件时

**执行步骤**：

```
1. 读取 02-Profile/my-cognitive-profile.md
2. 读取用户的原创文件
3. 分析：
   - 用户的这篇创作体现了什么语言风格？
   - 用户的推理模式是什么？
   - 用户的核心信念有变化吗？
4. 将分析结果反馈给用户
5. 根据用户确认，更新 my-cognitive-profile.md
```

**认知档案的更新原则**：

- 不是每次处理都更新
- 只有当发现**新的模式**或**信念变化**时才更新
- 更新时保留历史版本（在档案中记录"2026-04: 发现用户开始使用'结构'隐喻"）

## 五、链接治理规则

### 5.1  只用于已存在的卡片

与旧系统一致：禁止幽灵链接。

### 5.2 关系卡片的特殊规则

关系卡片可以在"涉及的概念"等部分列出**期望存在但尚未创建**的卡片，但**不使用 **，而是使用**加粗**：

```markdown
## 涉及的概念
- [[认知脚手架]] — 已存在
- **同伴互助** — 待创建（发现相关文章时创建）
```

### 5.3 冲突卡片的链接

冲突卡片必须链接到**涉及的具体概念卡片**：

```markdown
## 涉及的概念
- [[认知脚手架]]
- [[自我调节学习]]
```

## 六、安全与边界

### AI 禁止事项

❌ **不要**：
- 创建信息稀薄的空卡片
- 为每篇文章都创建新卡片
- 在 Inbox 中堆积已处理的原始文档
- 修改 02-Profile/my-cognitive-profile.md 未经用户确认
- 删除任何卡片（只能标记为废弃）
- 创建幽灵链接

### AI 必须事项

✅ **必须**：
- 优先丰富已有卡片，而非新建卡片
- 检测并记录冲突
- 维护关系卡片的完整性
- 处理完后归档原始文档
- 输出清晰的消化报告
- 所有AI生成的内容标注来源
