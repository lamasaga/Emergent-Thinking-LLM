# AGENTS.md — AI 操作手册（v3.3）

> **核心原则**：AI 不是来"分解文章"的，AI 是来"建模一个人的思想"的。
>
> 本项目已从读书笔记系统升级为**个人思想建模与外化系统**，目标是为未来 agent 的 GraphRAG 提供结构化的内容支持。

---

## 一、系统的核心逻辑

### 1.1 领域卡片是骨架

知识网络的组织方式不是文件夹，不是标签，而是**领域**。

- 顶级领域卡片（如"教学领域""认知领域"）是 AI 的**入口**。
- 领域卡片下涉及的实例卡片（概念、模型、方法、实体、冲突）是 AI **滋养**的对象。
- `01-Cards/_meta/ontology.md` 是当前元结构的**活契约**，AI 每次操作前必须读取。

### 1.2 AI 的渐进式阅读

AI **不允许**一次性读取全部卡片。必须遵循：

```
Step 1: 读取 01-Cards/_meta/ontology.md
Step 2: 读取 domains/ 下的全部顶级领域卡片
Step 3: 判断内容涉及哪些领域
Step 4: 读取相关领域卡片指向的实例卡片
Step 5: 进行比较、补充、冲突检测
Step 6: 更新领域卡片和 ontology.md（如有新发现）
```

### 1.3 内容是燃料，卡片是资产

**禁止行为**：
- ❌ 为每篇原始文档创建摘要文件
- ❌ 把内容拆成碎片分发到无数空卡片
- ❌ 在 Inbox 中堆积已处理的原始文档
- ❌ 创建幽灵链接
- ❌ 让卡片因无限引用而膨胀

**必须行为**：
- ✅ 用内容去**丰富已有卡片**
- ✅ 只在遇到**真正新的概念/模型/冲突**时才新建卡片
- ✅ 处理完后将原始文档移动到 03-Archive/
- ✅ 所有 AI 生成的内容标注来源
- ✅ 遵循实例引用上限：cards ≤ 6 条，profile ≤ 10 条

### 1.4 编译-缓冲-建构/消化流水线

项目工作流从「一次消化一篇文章」演进为：

```
00-Inbox/raw 文档
        │
        ▼
    /compile ──────► 05-Buffer/（原子化中间产物）
        │                       │
        │                       ▼
   03-Archive/            /digest（增量加工）
   （原始文档归档）         /construct（整体建构/重组）
                                │
                                ▼
                        01-Cards/ + 02-Profile/
```

- `/compile` 把 raw 文档**深度拆解**为 Buffer 中的高信息密度原子片段，保留原文摘录、精确来源与跨文档关联假设。
- `/digest` 只消费 Buffer 中尚未处理的片段，对现有卡片做增量更新。
- `/construct` 读取全部 Buffer 与全部卡片，通过主题聚类涌现领域、构建高质量概念网络与关系结构，并更新 Profile。
- `/refactor` 已合并进 `/construct`，原指令标记为 deprecated。

## 二、指令集

### 2.1 `/compile`（/编译）

**触发条件**：用户在 `00-Inbox/` 中放入 raw 文档后，主动说 `/compile`。

**执行步骤**：

```
1. 扫描 `00-Inbox/` 下全部文件，识别文档类型（零散文档 / PDF 图书 / 网页 / 播客转录等）。
2. 生成编译单元队列：
   a. **raw 文档**：`00-Inbox/` 中的单个独立文件（一篇论文、一本书、一个网页、一份报告）。
   b. **编译单元**：喂给 LLM 的一次文本块，受 `max-chars` 限制（默认 3 万等效中文字符）。
   c. 零散文档：整篇作为一个编译单元；若超过 3 万字则按语义段落切分。
   d. PDF 图书 / 长论文：优先按目录/章节切分为多个编译单元；无法提取目录时按页码边界切分；
      确保每个单元 ≤ 3 万字符，且尽量保留完整论证段落。
   e. 对每个编译单元记录精确来源锚点：文件路径、页码/章节、arXiv ID、DOI、URL、时间戳。
3. 深度原子化拆解（碎片数量按 **raw 文档** 计算，不是按编译单元）：
   - 每篇 raw 文档应产出 **3-8 个高信息密度碎片**；内容稀薄时允许更少，信息密集时长论文/专著允许显著更多。
   - 若多个短 raw 文档被合并到一个 3 万字的编译单元中，LLM 必须分别为每个 raw 文档独立产出碎片。
   - 对每个编译单元，LLM 从以下维度提取原子碎片，每个维度独立成文件：
   a. problem：核心问题、研究动机、试图回答什么问题。
   b. claim：作者核心主张、立场、结论性判断。
   c. model：提出的模型/框架/解释结构，含关键组件与它们的关系。
   d. method：方法/流程/实现细节，含输入、步骤、输出、关键超参数。
   e. evidence：实验/证据/评测，含数据集、指标、结果、与基线的对比。
   f. limit：限制、代价、未解决问题、适用范围边界。
   g. conflict：与现有工作、常识或本知识库中已有观点的张力/反对对象。
   h. connection：与 Inbox 中其他文档、或本知识库中已知概念的潜在关联假设。
   i. quote：值得保留的原文摘录（≤3 条/碎片），必须标注页码/章节。
   每个碎片选择一个最匹配的 type 写入 `05-Buffer/<type>/`。
4. 跨文档初步关联（同一批次内）：
   a. 在 Buffer 正文中用 **加粗** 标注与其他文档、其他碎片的潜在联系，例如「**DeepSeek-V3**」。
   b. 不使用 [[ ]]，避免幽灵链接。
   c. 在 compile 报告中列出高价值主题聚类与跨文档冲突假设。
5. 每轮结束后运行 `scripts/validate_buffer.py` 校验新生成 Buffer。
6. 每轮结束后运行 `scripts/check_digest_trigger.py` 检查 scratch Buffer 数量：
   - 默认阈值为 50，可通过环境变量 `BUFFER_DIGEST_THRESHOLD` 或命令行参数覆盖。
   - 若 scratch 数量 < 阈值：继续下一轮。
   - 若 scratch 数量 >= 阈值：输出「达到自动消化阈值，将自动进入 /digest」，
     并立即调用 `/digest` 流程。
7. 全部循环结束后输出总报告：
   - 原始文档数、编译单元数、批次数、Buffer 类型分布
   - 每篇 raw 文档的碎片数分布（平均 / 最大 / 最小）
   - 是否触发自动消化
   - 高价值主题聚类
   - 跨文档冲突/互补假设
   - 无法归类的片段数量与异常
8. 将原始文档移动到 `03-Archive/`。
```

**实现脚本**：`python scripts/compile_loop.py [--max-chars 30000] [--dry-run]`

**Buffer 质量标准**：
- 每个 Buffer 文件必须包含至少一条**原文摘录**或**精确来源锚点**（页码、章节、表格/图表编号）。
- 禁止仅写标题或一句话摘要；每个碎片必须能支撑后续卡片撰写。
- 同一文档的多个碎片之间应在正文中相互指涉，形成局部论证链。
- 来源标注格式统一：`《文件名》第 X 章 / 第 X 页` 或 `arXiv:xxxx.xxxxx §X`。

**约束**：
- `/compile` **只**生成 Buffer，不创建/修改 `01-Cards/` 或 `02-Profile/`。
- 不建立幽灵链接；Buffer 内提及外部概念用 **加粗**。
- 一个碎片一文件，正文严格单次写入。
- 对 raw 文档的拆解允许不完整；宁可少拆，不要硬造。
- 不因为类型目录存在就强行归类；无法判定 type 时放入 `05-Buffer/note/`。

### 2.2 `/construct`（/建构）

**触发条件**：
- 用户在 `05-Buffer/` 中积累了关键片段后，主动说 `/construct`。
- `/digest` 的输出报告指出存在需要整体重组的结构张力。
- 用户想从 Buffer 中重新涌现领域和思维结构。

`/refactor`（/重构）已合并进 `/construct`。当用户说 `/refactor` 时，AI 提示其已 deprecated，并询问是否以 `/construct` 继续执行。

**执行步骤**：

```
1. 读取 05-Buffer/ 中的全部文件（包括 status: scratch / digested / constructed）。
2. 读取 01-Cards/_meta/ontology.md。
3. 读取 domains/ 下的全部顶级领域卡片及范围内的实例卡片。
4. 主题聚类与概念涌现：
   a. 按高频概念、模型簇、方法簇、冲突簇对 Buffer 进行聚类。
   b. 识别跨 Buffer 重复出现、互为补充或互相对立的主题。
   c. 将聚类结果映射到现有卡片：可 enrich、可合并、可新建。
5. 识别核心主题 → 生成或调整顶级领域卡片（domains/*.md）。
   领域卡片必须说明：核心问题、边界、内在张力、下属核心实例卡片列表。
6. 识别跨领域的关键概念、实体、冲突、模型、方法 → 生成、合并、拆分或废弃实例卡片。
   每张核心卡片必须包含以下最小内容集合：
   a. 定义 / 核心思想（用一句话 + 一段展开）。
   b. 动机 / 解决的问题（为什么需要它）。
   c. 关键细节（公式、算法步骤、架构组件、因果链条）。
   d. 证据与代价（支持它的实验/案例、使用它要付出的代价）。
   e. 限制与边界（不适用场景、作者自己承认的局限）。
   f. 与其他卡片的关系（至少 2-4 条 outgoing relations）。
   g. 原文摘录（≤6 条，每条标注精确来源）。
7. 强制创建冲突卡片：
   a. 当 Buffer 中明确记录两个及以上对立主张、方法或理论时，必须创建 conflict 类型卡片。
   b. 冲突卡片必须链接到涉及的具体概念卡片（[[id|标题]]）。
   c. 冲突卡片需说明：对立双方、核心分歧点、各自的证据/代价、是否存在调和可能。
8. 建立关系网络：
   a. 在卡片 frontmatter relations 与正文中建立关系，优先使用 ontology.md 声明的类型。
   b. 单张卡片 outgoing relations 建议 3-6 条；领域卡片不受此限。
   c. 重点建立：extends、conflicts-with、supports、example-of、applies-to、influences、based-on、implements。
9. 根据涌现出的内容，设计或调整 01-Cards/ 的目录结构。
10. 识别结构信号（冗余、分裂、链接空洞、链接过载、类型错配、目录张力、Buffer 聚集、概念漂移、领域边界、幽灵链接等）。
11. 基于以上信号，提出 1-3 个整体结构方案，每个方案包含：
    a. 诊断：发现了什么组织问题。
    b. 动作：新建、合并、拆分、移动、重命名、调整类型、增删链接、新建/合并子领域等。
    c. 影响范围：涉及哪些卡片/领域/目录。
    d. 风险：可能丢失的信息、需要人工判断的地方。
12. 向用户呈现方案，等待用户选择或修改。未经用户确认，不执行任何写入。
13. 用户确认后执行重构：
    a. 合并卡片：保留一个主卡片，将被合并卡片的内容、来源、relations 迁移过去，原卡片 frontmatter 改为 status: deprecated，并在正文中说明合并目标。
    b. 拆分卡片：为新主题创建新卡片，拆分内容、来源、relations，更新原卡片。
    c. 调整链接：增删或修改 frontmatter relations 与正文 [[id|显示文本]]。
    d. 调整目录/领域：新建、重命名或合并领域目录；更新领域卡片；必要时更新 ontology.md。
    e. 修正类型：在确认后调整卡片的 type。
    f. 清理幽灵链接：修复或删除指向不存在或 deprecated 卡片的链接。
14. 基于种子/整合内容生成或更新 02-Profile/：
    a. 02-Profile/语言风格.md：表达 DNA、常用句式/修辞、术语偏好、原文摘录。
    b. 02-Profile/思维模型.md：心智模型（含三重验证）、决策启发式、推理模式、默认解决路径、智识谱系、内在张力、原文摘录。
    c. 02-Profile/核心信念.md：根本立场、价值观排序、反模式、诚实边界、原文摘录。
    d. 每条推断必须标注来源（文件名或具体引用）。
    e. 在 02-Profile/profile-index.md 中记录版本历史。
    f. 若某栏目种子内容不足，保留结构化空槽并标记为 awaiting-content，不编造内容。
15. 更新所有受影响卡片的 updated 字段。
16. 在 01-Cards/_meta/ontology.md 中记录结构规则的演化。
17. 将已整合进结构的 Buffer 文件 status 改为 constructed，并更新 updated。
18. 输出建构报告。
```

**卡片质量标准**：
- 每张新卡片必须达到「可学习、可参考」标准：一个不了解该主题的人阅读卡片后，能 grasp 核心思想并知道去哪里找原始出处。
- 禁止出现「待补充」「后续完善」等空槽；若某维度信息确实缺失，应明确标注「原文未提及」并说明缺失的是什么。
- 模型/方法类卡片必须包含具体细节（公式、步骤、参数、流程图描述），不能只有泛泛定义。
- 概念卡片应区分：作者原意、本知识库的解读、以及与其他概念的关系。

**约束**：
- `/construct` 可以反复调用，不限于项目初期。
- 不生成空卡片；不生成只有标题和一句话的卡片。
- `/construct` 基于 Buffer 生成或更新 02-Profile/ 时，条目 frontmatter 标记为 `status: seed`；当某 Profile 文件积累了 ≥3 个不同来源的支持证据，或用户明确确认时，可升级为 `growing`。
- 每个 Profile 文件的原文摘录总数 ≤ 10 条。
- `/construct` 禁止通过读取全部卡片来反推 Profile，只能基于 Buffer 和已生成的领域/实例卡片进行主题聚焦。
- 禁止删除任何卡片；合并后的原卡片标记为 `status: deprecated`。
- 合并/拆分卡片时，必须在相关卡片中记录历史沿革和原因。
- 所有新增、修改、删除的链接必须保证无幽灵链接。
- 若重构涉及 02-Profile/，需按 4.2 规则单独获得用户确认。

### 2.3 `/digest`（/消化）

**触发条件**：`05-Buffer/` 中有 `status: scratch` 的文件，或用户主动说 `/digest`。

**执行步骤**：

```
1. 读取 05-Buffer/ 中 status: scratch 的文件。
2. 读取 01-Cards/_meta/ontology.md。
3. 读取 domains/ 下的全部顶级领域卡片。
4. 判断新内容涉及哪些领域。
5. 读取相关领域卡片及其指向的实例卡片。
6. 进行比较分析：
   a. 概念/模型/方法是否已有卡片？
      → 有：丰富卡片，更新 sources 和 updated 字段
      → 无：判断是否重要且持久，决定是否新建
   b. 内容与现有卡片是否冲突？
      → 是：标记冲突，考虑新建冲突卡片
   c. 实体是否已有卡片？
      → 有：补充信息
      → 无：判断重要性后决定是否新建
7. 评估对 02-Profile/ 的影响：
   a. 读取当前 02-Profile/ 全部文件。
   b. 对比新内容与现有条目，检测以下变化：
      - 语言风格：新的句式/修辞、词汇特征、幽默方式、确定性表达、引用习惯
        → 确认后追加到 02-Profile/语言风格.md
      - 思维模型：新的心智模型、决策启发式、智识谱系影响、内在张力
        → 确认后追加到 02-Profile/思维模型.md
        → 新模型必须通过三重验证（跨域复现、生成力、排他性）中至少两项
      - 核心信念：新的根本立场、价值观排序变化、反模式、诚实边界
        → 确认后追加/修订 02-Profile/核心信念.md
   c. 当某 Profile 文件积累了 ≥3 个不同来源的支持证据，或用户明确确认时，
      将 frontmatter status 从 seed 升级为 growing。
   d. 若原文摘录超过 10 条上限，向用户报告并请求替换决策。
   e. 在 02-Profile/profile-index.md 中追加版本历史记录。
   f. 若无变化：在消化报告中说明「未检测到 Profile 变化」。
8. 检查目录结构是否需要调整：
   a. 现有子目录能否容纳新卡片？
   b. 是否需要新建子目录？
   c. 是否需要合并/重命名旧目录？
   d. 更新 ontology.md 记录结构变更
9. 更新领域卡片。
10. 将本次读取的 scratch Buffer 文件 status 改为 digested，并更新 updated。
11. 输出消化报告。
```

**约束**：
- `/digest` **不删除、不移动** Buffer 文件。
- AI 更新 02-Profile/ 前必须向用户报告变更点，经确认后写入。
- 遵循实例引用上限：cards ≤ 6 条，profile ≤ 10 条。
- 禁止幽灵链接。

### 2.4 `/analyze`（/分析）

**触发条件**：用户对一个具体问题说 `/analyze`。

**执行步骤**：

```
1. 解析用户提出的问题。
2. 读取 02-Profile/ 下的全部档案文件。
3. 读取 01-Cards/_meta/ontology.md。
4. 读取 domains/ 下可能与问题相关的领域卡片。
5. 根据领域卡片的指向，读取相关实例卡片。
6. 在知识库中检索与问题相关的概念、模型、冲突、实体。
7. （可选）当问题明显涉及近期原始动机时，读取 05-Buffer/ 中的 scratch 文件作为补充上下文。
8. 组装分析上下文包（Context Package），写入 04-OutBox/YYYYMMDD-主题-context.md：
   a. 用户透镜：从 Profile 中蒸馏与问题相关的心智模型、核心信念、语言风格提示、诚实边界。
   b. 相关领域：列出相关领域卡片及简要说明。
   c. 相关卡片：按类型（概念、模型、方法、实体、冲突等）分组，提供摘要、状态、1-3 条关键摘录、相关关系。
   d. 关系子图：抽取相关卡片之间的显式关系。
   e. 冲突与张力的显式标注。
   f. 假设链（可选）：当问题涉及 ≥3 个卡片、≥2 个领域、关键证据不足，或用户要求详细分析时生成；否则标注「未触发」。
   g. 盲区与假设：列出知识库缺口、需要依赖通用推理的地方、Profile 可能过时的维度。
   h. 来源：列出原始文档/卡片来源。
9. 基于上下文包生成回答，输出到 04-OutBox/YYYYMMDD-主题.md：
   a. 只读取 Context Package，不再回头读取原始卡片。
   b. 用用户的思维模型组织论证。
   c. 体现用户的语言风格。
   d. 尊重用户的核心信念。
   e. 引用相关卡片（[[id|标题]]）。
   f. 指出知识库中的冲突或张力。
10. 在 OutBox 文件顶部标注问题来源、引用卡片 id 列表、分析时间、关联的 Context Package 路径。
11. 输出简要报告。
```

## 三、卡片规范

### 3.1 YAML frontmatter

```yaml
---
id: "cognitive-scaffolding"
title: "认知脚手架"
type: "notion"
created: "2026-07-07"
updated: "2026-07-07"
sources:
  - "来自《某书》第 3 章"
relations:
  - target: "zone-of-proximal-development"
    type: "extends"
    note: "脚手架作用在最近发展区内"
tags:
  - "教学"
status: "seed"   # seed | growing | mature | deprecated
---
```

### 3.2 卡片类型

由 `ontology.md` 声明。初始集合：

| type | 定义 | 回答的问题 |
|------|------|-----------|
| `domain` | 领域卡片 | 「这是哪个思想领域？」 |
| `notion` | 观念、立场、信条 | 「我认为/怎么看这件事？」 |
| `principle` | 原则、原理、规律 | 「什么必须被遵守？为什么会这样？」 |
| `phenomenon` | 现象、模式、趋势 | 「发生了什么？呈现什么模式？」 |
| `entity` | 有明确边界的实体对象 | 「这个具体对象是什么？」 |
| `group` | 多个实体构成的群体 | 「这是哪一类人/事物的群体？」 |
| `model` | 模型、框架、解释结构 | 「这个复杂事物怎么理解？」 |
| `method` | 方法、技巧、流程 | 「怎么做才能达到目标？」 |
| `conflict` | 冲突、张力、两难 | 「这里的根本矛盾是什么？」 |
| `note` | 暂时无法归类的想法 | 「这个想法还没成熟」 |

**类型边界判定**：当一张卡片可能属于多个类型时，按以下优先级选择主类型：

1. 约束判断或行动 → `principle`
2. 解释复杂事物结构 → `model`
3. 提供可执行步骤 → `method`
4. 描述可观察模式 → `phenomenon`
5. 表达个人立场 → `notion`
6. 有明确边界的独立对象 → `entity`
7. 多个对象构成的群体 → `group`
8. 记录根本对立 → `conflict`

### 3.3 目录组织

- `01-Cards/domains/`：领域卡片。
- `01-Cards/[领域目录]/`：该领域下的实例卡片，按领域组织，不按类型组织。
- 同一领域目录下可混合存放 `notion`、`principle`、`method` 等多种类型卡片。
- `05-Buffer/`：由 `/compile` 生成的原子化中间产物，按 ontology type 分目录，文件命名 `YYYY-MM-DD-HHMMSS-关键词.md`。

### 3.4 实例引用上限

- `01-Cards/` 卡片：原始资料实例 ≤ **6 条**。
- `02-Profile/` 档案：原文摘录 ≤ **10 条**。
- 超过上限时，AI 比较新旧内容，用更优质的替换。

### 3.5 链接治理

#### 3.5.1 只用于已存在的卡片

最高标准：**禁止幽灵链接**。`[[id|显示文本]]` 或 `[[id]]` 必须指向真实存在的卡片。

#### 3.5.2 Buffer 文件的链接规则

`05-Buffer/` 中的中间产物**不使用 `[[ ]]`**。提及已有卡片时用 **加粗** 或纯文本，例如「**认知脚手架**」。这样可以让 Buffer 保持草稿身份，避免链接治理负担。

#### 3.5.3 领域卡片的特殊规则

领域卡片可以在「涉及的概念」等部分列出**期望存在但尚未创建**的卡片，但**不使用 `[[ ]]`**，而是使用**加粗**：

```markdown
## 涉及的概念
- [[cognitive-scaffolding]] — 已存在
- **同伴互助** — 待创建（发现相关文章时创建）
```

待出现与之对应的卡片时，修正并创建链接。

#### 3.5.4 冲突卡片的链接

冲突卡片必须链接到**涉及的具体概念卡片**：

```markdown
## 涉及的概念
- [[cognitive-scaffolding]]
- [[self-regulated-learning]]
```

#### 3.5.5 高相关卡片之间的链接建立

`/digest` 负责在日常消化中维护卡片与领域之间的关系，并在高度相关的实例卡片之间建立显式链接；`/construct` 则负责当结构张力累积时，主动涌现新的组织模式、调整链接、合并或拆分卡片。

**判断高相关的信号**：
- 在同一篇原始文档的同一论点中被共同提及；
- 共享多个相同来源；
- 属于同一领域且语义上互补、对立或扩展；
- 一方是另一方的具体例子、应用场景或反对对象。

**建立链接的方法**：
1. 在两张卡片的 YAML frontmatter `relations` 中分别补充指向对方的关系，例如：
   ```yaml
   relations:
     - target: "self-regulated-learning"
       type: "supports"
       note: "自我调节学习是脚手架拆除后的目标"
   ```
2. 在卡片正文中，当上下文自然提到对方时，插入 `[[id|显示文本]]` 链接。
3. 关系类型优先从 `ontology.md` 中声明的类型选择（如 `extends`、`conflicts-with`、`supports`、`example-of`、`applies-to`、`influences`），若无合适类型则向用户报告，经确认后再新增。

**边界**：
- 避免为了链接而链接；只有确实存在思想关联时才建立。
- 单张卡片的 outgoing relations 建议不超过 6 条，防止卡片因无限引用而膨胀。
- 领域卡片不受上一条约束，因为领域卡片要起到统摄下属卡片的作用。
- 若建立关系后发现两张卡片应合并，应向用户报告，而不是直接合并。

## 四、02-Profile/ 使用规则

### 4.1 档案组成

- `profile-index.md`：总览与版本历史
- `语言风格.md`：表达 DNA（句式、词汇、节奏、幽默、确定性、引用）+ 常用句式/修辞、术语偏好
- `思维模型.md`：心智模型（含三重验证）、决策启发式、推理模式、默认解决路径、智识谱系、内在张力
- `核心信念.md`：根本立场、价值观排序、反模式、诚实边界

### 4.2 更新原则

- `/construct` 在建构阶段基于 Buffer 生成或更新 Profile，frontmatter 标记为 `status: seed`。
- `/digest` 不每次消化都更新，只在发现新模式或信念变化时才更新。
- AI 更新前必须向用户报告变更点，经确认后写入。
- 每次更新在 `profile-index.md` 的「Profile 演化日志」中追加记录。
- 当某 Profile 文件积累了 ≥3 个不同来源的支持证据，或用户明确确认时，可将 `status` 从 seed 升级为 growing。

## 五、安全与边界

### AI 禁止事项

❌ **不要**：
- 创建信息稀薄的空卡片或 Buffer 碎片
- 为每篇文档都创建新卡片
- 在 Inbox 中堆积已处理的原始文档
- 未经用户确认修改 02-Profile/
- 删除任何卡片（只能标记为 deprecated）
- 创建幽灵链接
- 让卡片因无限引用而膨胀
- 在 Buffer 或卡片中使用无来源的推断

### AI 必须事项

✅ **必须**：
- 优先丰富已有卡片，而非新建卡片
- 检测并记录冲突
- 维护领域卡片的完整性
- 处理完后归档原始文档
- 输出清晰的编译/建构/消化/分析报告
- 所有 AI 生成的内容标注来源
- 遵循实例引用上限
- `/compile` 保留原文摘录与精确来源锚点
- `/construct` 生成的卡片达到「可学习、可参考」标准

## 六、Hooks 与自动化

为了把规则明确的确定性操作从 LLM 中抽离，项目使用 hooks 处理校验与执行。LLM 仍负责语义判断、涌现模式识别和需要人工确认的创作型决策。

### 6.1 启用方式

```bash
# 让 Git 使用项目内的 hooks 目录
git config core.hooksPath hooks

# 提交前会自动运行卡片校验
```

### 6.2 当前 hooks

#### `scripts/validate_cards.py` + `hooks/pre-commit`

提交前自动运行，检查：

- `01-Cards/` 卡片：
  - frontmatter 必填字段完整性（`id`, `title`, `type`, `created`, `updated`, `status`）
  - `id` 唯一性
  - `type` 有效性
  - 正文与 `relations` 中的链接是否指向真实存在的卡片（**幽灵链接检测**）
  - `relations` 数量是否超过建议上限 6
  - `sources` 列表项数量是否超过上限 6
  - 正文 `## 实例` 部分引用数量是否超过上限 6
  - 是否存在未被任何领域卡片引用的 **orphan 实例卡片**
- `05-Buffer/` 中间产物：
  - frontmatter 必填字段（`title`, `type`, `created`, `updated`, `status`）
  - `status` 取值有效性
  - `type` 与所在子目录名称一致
  - 文件名符合 `YYYY-MM-DD-HHMMSS-关键词.md` 格式
  - 正文不含 `[[ ]]` 链接
  - frontmatter 不含 `id` 或 `relations`
  - `scratch` 文件数量是否超过建议阈值

校验发现错误时阻止提交；发现警告时允许提交，但会提示用户。

### 6.3 hooks 与指令的协作

| 指令 | hooks 承担的确定性工作 | LLM 承担的语义工作 |
|---|---|---|
| `/compile` | 扫描与分类、编译单元生成、批次组织、调用 `scripts/validate_buffer.py` 与 `scripts/check_digest_trigger.py`、原始文档归档移动 | raw 文本的**深度原子化拆解**（problem/claim/model/method/evidence/limit/conflict/connection/quote）、跨文档关联假设、暂定类型判断、决定是否自动触发 /digest |
| `/construct` | 目录创建、frontmatter 模板校验、id 唯一性检查、结构信号扫描、变更清单格式化、卡片质量校验 | Buffer 主题聚类、高质量概念卡片涌现、关系网络构建、冲突卡片强制创建、领域设计、整体重组方案、Profile 更新 |
| `/digest` | frontmatter 补全、updated 字段更新、实例/引用上限检查 | 概念匹配、冲突检测、新建/丰富判断、Profile 变更报告 |
| `/analyze` | Context Package 格式校验、输出文件命名、引用列表提取 | 推理、论证、风格化表达 |

### 6.4 添加新 hooks

新增 hook 时应遵循：

1. 只处理有明确规则的任务。
2. 不替代 LLM 的语义判断。
3. 失败时给出清晰的可操作提示。
4. 更新本手册的 hooks 说明。
