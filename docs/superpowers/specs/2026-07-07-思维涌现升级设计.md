# 思维涌现org 升级设计规格 v1.0

> **目标**：将本项目从一个「读书管理笔记项目」升级为「特定用户的个人思想建模与外化系统」，并为未来 agent 的 GraphRAG 提供结构化的内容支持。

---

## 1. 背景与目标

### 1.1 当前状态

项目当前以「关系卡片」为骨架，通过 `/digest` 消化外来文章，滋养 `01-Cards/` 下的概念、实体、关系、冲突四类卡片，并在 `02-Profile/` 中维护单一认知档案。

### 1.2 升级目标

1. **个人思想建模**：项目不再只是管理读书笔记，而是持续建模某个特定用户的思想结构。
2. **GraphRAG 内容支持**：卡片格式与目录结构需方便未来 agent 解析为知识图。
3. **动态目录结构**：`01-Cards/` 下的子目录不由用户预先固定，而由 LLM 根据消化内容涌现、调整。
4. **统一入口**：`00-Inbox/` 中不再区分外来文章与自己的思考，所有内容都参与 `/digest` 和 Profile 维护。
5. **风格文件夹升级**：`02-Profile/` 需要更好的组织形式、可读性与可用性。
6. **新增 `/analyze` 与 `04-OutBox/`**：用户可以要求 AI 基于知识库和认知档案分析问题，并输出优质回答。

---

## 2. 核心设计原则

### 2.1 领域驱动，而非关系驱动

原设计的骨架是「关系卡片」，容易让 AI 把一切都理解成关系。升级后，骨架改为「领域卡片」（`domains/`）。领域是用户思想中的核心主题空间，关系、概念、冲突等都在领域内生长。

### 2.2 元结构约束 + 实例动态

- **元结构**：由 `01-Cards/_meta/ontology.md` 声明，约束顶层领域、允许的卡片类型、目录组织原则、命名约定。
- **实例动态**：具体子目录、概念卡片、模型卡片等由 `/construct` 和 `/digest` 根据内容涌现。

### 2.3 卡片是资产，Inbox 是燃料

原始文档（无论是外来文章还是自己的思考）在消化后进入 `03-Archive/`。卡片才是持续积累的知识资产。

### 2.4 内容必须机器可读

所有卡片和档案统一使用 YAML frontmatter，便于未来 agent 做 GraphRAG 索引。

### 2.5 禁止无限膨胀

- `01-Cards/` 中每张卡片的原始资料实例展示 **≤ 4 条**。
- `02-Profile/` 中每个档案的原文摘录 **≤ 10 条**。
- 超过上限时，AI 需比较新旧内容，用更优质的替换现有条目。

## 3. 升级后的目录结构

```
思维涌现org/
├── 00-Inbox/                        ← 所有原始内容的入口
│   └── （用户直接放入文章或思考，不再分子文件夹）
│
├── 01-Cards/                        ← 核心知识区
│   ├── _meta/
│   │   ├── ontology.md              ← 当前元结构声明
│   │   └── schema-version.md        ← schema 版本与变更日志
│   │
│   ├── domains/                     ← 领域卡片 = 骨架/目录
│   │   ├── 教学领域.md
│   │   ├── 认知领域.md
│   │   └── ...（由内容涌现）
│   │
│   └── [动态目录]/                  ← 由 /construct 和 /digest 涌现的实例目录
│       └── *.md                     ← 概念、实体、冲突、模型、方法等卡片
│
├── 02-Profile/                      ← 风格文件夹 / 认知档案
│   ├── profile-index.md             ← 档案总览与版本历史

│   └── 核心信念.md

│
└── 04-OutBox/                       ← /analyze 输出目录
    └── YYYYMMDD-主题.md
```

## 4. 卡片规范

### 4.1 YAML frontmatter 模板

```yaml
---
id: "cognitive-scaffolding"
title: "认知脚手架"
type: "concept"                    # 见 4.2 类型约束
created: "2026-07-07"
updated: "2026-07-07"
sources:
  - "来自《某书》第 3 章"
  - "来自 00-Inbox/我的思考草稿.md"
relations:
  - target: "zone-of-proximal-development"
    type: "extends"
    note: "脚手架作用在最近发展区内"
  - target: "teaching-domain"
    type: "belongs-to"
    note: "属于教学领域的核心概念"
tags:
  - "教学"
  - "支持"
status: "seed"                     # seed | growing | mature | deprecated
---
```

### 4.2 类型约束

`type` 字段允许的值由 `ontology.md` 声明。初始建议集合：

- `domain`：领域卡片，骨架。
- `concept`：抽象概念。
- `entity`：具体的人、机构、作品、理论。
- `model`：思维模型、解释框架。
- `method`：方法、技巧、流程。
- `conflict`：冲突与张力的记录。
- `note`：暂时无法归类的想法。

### 4.3 ID 与文件名约定

- `id`：全局唯一，使用 `kebab-case` 或 `camelCase`，建议用 `kebab-case`。
- 文件名：与 `id` 一致，例如 `cognitive-scaffolding.md`。
- 中文标题放在 `title` 字段和 `# 标题` 中。

### 4.4 链接治理

- 正文中的 `[[id|显示文本]]` 或 `[[id]]` 必须指向真实存在的卡片。
- 领域卡片可以在「涉及的实例」部分用 **粗体** 列出待创建的概念，但不使用 `[[ ]]`，避免幽灵链接。

## 5. 元结构声明：`_meta/ontology.md`

`ontology.md` 是当前知识结构的「活契约」，LLM 每次操作前都应读取。内容建议包括：

```markdown
# 知识库元结构

## 当前领域（顶层骨架）
- [[teaching-domain|教学领域]]
- [[cognitive-domain|认知领域]]
- ...

## 允许的卡片类型
- domain
- concept
- entity
- model
- method
- conflict
- note

## 目录组织原则
- 领域卡片统一放在 `domains/`。
- 实例卡片按 `/construct` 或 `/digest` 决定的子目录存放。
- 子目录命名使用 `kebab-case`，与内容主题一致。

## 允许的 relation 类型
- belongs-to
- extends
- conflicts-with
- supports
- example-of
- applies-to
- influences

## 命名约定
- id：`kebab-case`
- 文件名：与 id 一致
- 标签：`kebab-case`，中文也可
```

## 6. `/construct`（/建构）指令

### 6.1 触发条件

用户在 `00-Inbox/` 中放入若干篇最关键的「种子文章/思考」后，主动说 `/construct` 或 `/建构`。

### 6.2 执行步骤

```
1. 读取 00-Inbox/ 下全部文件。
2. 识别用户反复关心的核心主题，生成顶级领域卡片（domains/*.md）。
3. 识别跨领域的关键概念、实体、冲突、模型，生成实例卡片。
4. 根据涌现出的内容，设计 01-Cards/ 的具体目录结构。
5. 撰写 01-Cards/_meta/ontology.md，声明当前元结构。
6. 为每张生成的卡片写入 YAML frontmatter。
7. 建立领域卡片与实例卡片之间的显式 relations。
8. 输出建构报告：生成内容、目录结构设计理由、下一步建议。
9. 将已用于建构的种子文档移动到 03-Archive/。
```

### 6.3 约束

- `/construct` 只跑一次或少数几次，建立基础骨架。
- 不生成空卡片，每张卡片必须有实质性内容。
- 用户可以在报告后要求调整领域、合并概念或修改目录。

## 7. `/digest`（/消化）指令

### 7.1 触发条件

`00-Inbox/` 中有新文件。不再区分外来文章与自己的思考。

### 7.2 执行步骤

```
1. 读取 00-Inbox/ 中的新文件。
2. 读取 01-Cards/_meta/ontology.md。
3. 读取 domains/ 下的全部顶级领域卡片。
4. 判断新内容涉及哪些领域。
5. 读取相关领域卡片及其指向的实例卡片。
6. 进行比较分析：
   a. 概念/模型/方法是否已有卡片？
      → 有：丰富卡片，更新 sources 和 updated 字段
      → 无：判断是否重要且持久，决定是否新建
   b. 内容与现有卡片是否冲突？
      → 是：标记冲突，考虑新建冲突卡片
   c. 实体是否已有卡片？
      → 有：补充信息
      → 无：判断重要性后决定是否新建
7. 评估对 02-Profile/ 的影响：
   - 是否体现新的语言风格？
   - 是否使用新的思维模型？
   - 是否表达新的核心信念或立场变化？
   → 若有：向用户报告，经确认后更新 Profile
   → 若无：在消化报告中说明「未检测到 Profile 变化」
8. 检查目录结构是否需要调整：
   a. 现有子目录能否容纳新卡片？
   b. 是否需要新建子目录？
   c. 是否需要合并/重命名旧目录？
   d. 更新 _meta/ontology.md 记录结构变更
9. 更新领域卡片（添加新链接、调整涉及实例列表）。
10. 将原始文档移动到 03-Archive/。
11. 输出消化报告。
```

### 7.3 实例引用上限

- 每张卡片的原始资料实例展示 **≤ 4 条**。
- 超过 4 条后，AI 比较新旧内容，用更优质的替换。
- 被替换的旧实例不删除，保留在 `03-Archive/` 的原始文档中。

## 8. `/relate`（/关联）指令

### 8.1 触发条件

新建卡片后，或用户发现新的关联模式时。

### 8.2 执行步骤

```
1. 读取新卡片或目标卡片。
2. 判断该卡片属于哪些领域。
3. 读取对应领域卡片。
4. 在领域卡片的「涉及实例」部分添加链接或粗体待创建项。
5. 在实例卡片的 frontmatter relations 中补充反向关系。
6. 检查是否需要新建子领域。
```

### 8.3 说明

`/relate` 可以是独立指令，也可以作为 `/digest` 的内部步骤。为简化操作，建议默认由 `/digest` 自动完成，只在需要专门整理关联时调用 `/relate`。

## 9. `/analyze`（/分析）指令与 `04-OutBox/`

### 9.1 触发条件

用户对一个具体问题说 `/analyze` 或 `/分析`。

### 9.2 执行步骤

```
1. 解析用户提出的问题。
2. 读取 02-Profile/ 下的全部档案文件。
3. 读取 01-Cards/_meta/ontology.md。
4. 读取 domains/ 下可能与问题相关的领域卡片。
5. 根据领域卡片的指向，读取相关实例卡片。
6. 在知识库中检索与问题相关的概念、模型、冲突、实体。
7. 综合分析：
   a. 用用户的思维模型组织论证
   b. 体现用户的语言风格
   c. 尊重用户的核心信念
   d. 引用相关卡片（[[id|标题]]）
   e. 指出知识库中的冲突或张力
8. 生成优质回答，输出到 04-OutBox/YYYYMMDD-主题.md。
9. 在 OutBox 文件顶部标注问题来源、引用卡片 id 列表、分析时间。
10. 输出简要报告，告知用户文件已生成。
```

### 9.3 输出文件规范

**命名**：

```
04-OutBox/YYYYMMDD-主题.md
```

如果当天同主题多次分析，后缀加序号：

```
04-OutBox/YYYYMMDD-主题-01.md
04-OutBox/YYYYMMDD-主题-02.md
```

**frontmatter**：

```yaml
---
date: "2026-07-07"
type: "analysis-output"
question: "用户提出的原始问题"
sources:
  - "引用卡片 id1"
  - "引用卡片 id2"
profile-influenced: true
---
```

## 10. `02-Profile/` 风格文件夹组织

### 10.1 文件结构

```
02-Profile/
├── profile-index.md          ← 档案总览、版本历史、使用说明
├── 语言风格.md
├── 思维模型.md
└── 核心信念.md
```

### 10.2 各文件内容

**`profile-index.md`**

- 档案目的说明
- 版本历史（每次更新的时间、触发原因、变更摘要）
- 三张档案的读取顺序与使用方式

**`语言风格.md`**

- 常用句式与修辞
- 术语偏好
- 段落节奏
- 引用习惯
- 原文摘录（≤ 10 条）

**`思维模型.md`**

- 常用分析框架
- 推理模式
- 解决问题的默认路径
- 原文摘录（≤ 10 条）

**`核心信念.md`**

- 对学习、认知、教育、知识、创作等主题的根本立场
- 明确反对什么
- 认为什么最重要
- 原文摘录（≤ 10 条）

### 10.3 更新原则

- 不每次消化都更新。
- 只有发现新模式或信念变化时才更新。
- AI 更新前必须向用户报告变更点，经确认后写入。
- 每次更新在 `profile-index.md` 中记录版本历史。

## 11. GraphRAG 准备

本次升级为未来 GraphRAG agent 做以下准备：

1. **统一 YAML frontmatter**：所有卡片和档案都有结构化元数据。
2. **显式 `relations` 字段**：frontmatter 中声明卡片之间的关系，可直接构建图边。
3. **全局 `id` 体系**：避免依赖文件名或标题。
4. **`_meta/ontology.md`**：作为图的 schema。
5. **禁止幽灵链接**：保证 `[[id]]` 都指向真实节点。
6. **`profile-index.md` 作为用户画像节点**：未来可影响检索权重和生成风格。

**暂不做**：自动生成 embedding、外部向量数据库、RDF/OWL 本体。这些留给未来 agent 自行实现。

## 13. 实施边界

- 本次只改文档和目录结构，**不生成真实卡片**（除非用户下一步提供种子文章执行 `/construct`）。
