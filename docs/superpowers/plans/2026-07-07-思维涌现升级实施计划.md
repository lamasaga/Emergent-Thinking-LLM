> 本文件是当前知识结构的「活契约」。AI 在每次 `/construct`、`/digest`、`/analyze` 前都应读取它。

## 当前领域（顶层骨架）

## 允许的卡片类型

- `domain`：领域卡片，骨架。
- `concept`：抽象概念。
- `entity`：具体的人、机构、作品、理论。
- `model`：思维模型、解释框架。
- `method`：方法、技巧、流程。
- `conflict`：冲突与张力的记录。
- `note`：暂时无法归类的想法。

## 目录组织原则

- 领域卡片统一放在 `01-Cards/domains/`。
- 实例卡片按 `/construct` 或 `/digest` 决定的子目录存放。
- 子目录命名使用 `kebab-case`，与内容主题一致。

## 允许的 relation 类型

- `belongs-to`：属于某个领域或概念。
- `extends`：扩展、深化某个概念。
- `conflicts-with`：与某个概念冲突。
- `supports`：支持某个概念或立场。
- `example-of`：某个概念的具体例子。
- `applies-to`：应用于某个场景或问题。
- `influences`：影响某个概念或领域。

## 命名约定

- `id`：`kebab-case`，全局唯一。
- 文件名：与 `id` 一致，例如 `cognitive-scaffolding.md`。
- 中文标题放在 `title` 字段和 `# 标题` 中。
- 标签：`kebab-case` 或中文，小写。

## 实例引用上限

- `01-Cards/` 卡片：原始资料实例 ≤ 4 条。
- `02-Profile/` 档案：原文摘录 ≤ 10 条。

# AGENTS.md — AI 操作手册（v3.0）

> **核心原则**：AI 不是来"分解文章"的，AI 是来"建模一个人的思想"的。
>
> 本项目已从读书笔记系统升级为**个人思想建模与外化系统**，目标是为未来 agent 的 GraphRAG 提供结构化的内容支持。

## 一、系统的核心逻辑

### 1.1 领域卡片是骨架

知识网络的组织方式不是文件夹，不是标签，而是**领域**。

- 顶级领域卡片（如"教学领域""认知领域"）是 AI 的**入口**。
- 领域卡片下涉及的实例卡片（概念、模型、方法、实体、冲突）是 AI **滋养**的对象。
- `01-Cards/_meta/ontology.md` 是当前元结构的**活契约**，AI 每次操作前必须读取。

### 1.2 AI 的渐进式阅读

AI **不允许**一次性读取全部卡片。必须遵循：

```
Step 1: 读取 01-Cards/_meta/ontology.md
Step 2: 读取 domains/ 下的全部顶级领域卡片
Step 3: 判断内容涉及哪些领域
Step 4: 读取相关领域卡片指向的实例卡片
Step 5: 进行比较、补充、冲突检测
Step 6: 更新领域卡片和 ontology.md（如有新发现）
```

### 1.3 内容是燃料，卡片是资产

**禁止行为**：
- ❌ 为每篇原始文档创建摘要文件
- ❌ 把内容拆成碎片分发到无数空卡片
- ❌ 在 Inbox 中堆积已处理的原始文档
- ❌ 创建幽灵链接
- ❌ 让卡片因无限引用而膨胀

**必须行为**：
- ✅ 用内容去**丰富已有卡片**
- ✅ 只在遇到**真正新的概念/模型/冲突**时才新建卡片
- ✅ 处理完后将原始文档移动到 03-Archive/
- ✅ 所有 AI 生成的内容标注来源
- ✅ 遵循实例引用上限：cards ≤ 4 条，profile ≤ 10 条

## 二、指令集

### 2.1 `/construct`（/建构）

**触发条件**：用户在 `00-Inbox/` 中放入若干篇最关键的"种子文章/思考"后，主动说 `/construct`。

**执行步骤**：

```
1. 读取 00-Inbox/ 下全部文件。
2. 读取 01-Cards/_meta/ontology.md。
3. 识别用户反复关心的核心主题 → 生成顶级领域卡片（domains/*.md）。
4. 识别跨领域的关键概念、实体、冲突、模型 → 生成实例卡片。
5. 根据涌现出的内容，设计 01-Cards/ 的目录结构。
6. 更新 ontology.md，声明当前元结构。
7. 为每张卡片写入 YAML frontmatter。
8. 建立领域卡片与实例卡片之间的显式 relations。
9. 输出建构报告。
10. 将种子文档移动到 03-Archive/。
```

**约束**：
- `/construct` 只跑一次或少数几次，建立基础骨架。
- 不生成空卡片。
- 用户可在报告后要求调整。

### 2.2 `/digest`（/消化）

**触发条件**：`00-Inbox/` 中有新文件。不再区分外来文章与自己的思考。

**执行步骤**：

```
1. 读取 00-Inbox/ 中的新文件。
2. 读取 01-Cards/_meta/ontology.md。
3. 读取 domains/ 下的全部顶级领域卡片。
4. 判断新内容涉及哪些领域。
5. 读取相关领域卡片及其指向的实例卡片。
6. 进行比较分析：
   a. 概念/模型/方法是否已有卡片？
      → 有：丰富卡片，更新 sources 和 updated 字段
      → 无：判断是否重要且持久，决定是否新建
   b. 内容与现有卡片是否冲突？
      → 是：标记冲突，考虑新建冲突卡片
   c. 实体是否已有卡片？
      → 有：补充信息
      → 无：判断重要性后决定是否新建
7. 评估对 02-Profile/ 的影响：
   - 是否体现新的语言风格？
   - 是否使用新的思维模型？
   - 是否表达新的核心信念或立场变化？
   → 若有：向用户报告，经确认后更新 Profile
   → 若无：在消化报告中说明「未检测到 Profile 变化」
8. 检查目录结构是否需要调整：
   a. 现有子目录能否容纳新卡片？
   b. 是否需要新建子目录？
   c. 是否需要合并/重命名旧目录？
   d. 更新 ontology.md 记录结构变更
9. 更新领域卡片。
10. 将原始文档移动到 03-Archive/。
11. 输出消化报告。
```

### 2.3 `/relate`（/关联）

**触发条件**：新建卡片后，或用户发现新的关联模式时。

**执行步骤**：

```
1. 读取目标卡片。
2. 判断该卡片属于哪些领域。
3. 读取对应领域卡片。
4. 在领域卡片的「涉及实例」部分添加链接或粗体待创建项。
5. 在实例卡片的 frontmatter relations 中补充反向关系。
6. 检查是否需要新建子领域。
```

**说明**：默认由 `/digest` 自动完成，只在需要专门整理关联时独立触发。

### 2.4 `/analyze`（/分析）

**触发条件**：用户对一个具体问题说 `/analyze`。

**执行步骤**：

```
1. 解析用户提出的问题。
2. 读取 02-Profile/ 下的全部档案文件。
3. 读取 01-Cards/_meta/ontology.md。
4. 读取 domains/ 下可能与问题相关的领域卡片。
5. 根据领域卡片的指向，读取相关实例卡片。
6. 在知识库中检索与问题相关的概念、模型、冲突、实体。
7. 综合分析：
   a. 用用户的思维模型组织论证
   b. 体现用户的语言风格
   c. 尊重用户的核心信念
   d. 引用相关卡片（[[id|标题]]）
   e. 指出知识库中的冲突或张力
8. 生成优质回答，输出到 04-OutBox/YYYYMMDD-主题.md。
9. 在 OutBox 文件顶部标注问题来源、引用卡片 id 列表、分析时间。
10. 输出简要报告。
```

## 三、卡片规范

### 3.1 YAML frontmatter

```yaml
---
id: "cognitive-scaffolding"
title: "认知脚手架"
type: "concept"
created: "2026-07-07"
updated: "2026-07-07"

  - "来自《某书》第 3 章"
relations:
  - target: "zone-of-proximal-development"
    type: "extends"
    note: "脚手架作用在最近发展区内"
tags:
  - "教学"

---
```

### 3.2 卡片类型

由 `ontology.md` 声明。初始集合：

- `domain`：领域卡片
- `concept`：概念卡片
- `entity`：实体卡片
- `model`：模型卡片
- `method`：方法卡片
- `conflict`：冲突卡片
- `note`：笔记卡片

### 3.3 实例引用上限

- `01-Cards/` 卡片：原始资料实例 ≤ **4 条**。
- `02-Profile/` 档案：原文摘录 ≤ **10 条**。
- 超过上限时，AI 比较新旧内容，用更优质的替换。

### 3.4 链接治理

- `[[id|显示文本]]` 或 `[[id]]` 必须指向真实存在的卡片。
- 领域卡片中待创建的概念使用 **粗体**，不使用 `[[ ]]`。

## 核心设计哲学

### 思想是被建模的，不是被堆放的

外来文章和你自己的思考不再是孤立的文档，而是被"消化"进已有卡片的燃料。

- 卡片数量增长缓慢
- 每张卡片内容不断丰富
- 原始文档消化后进入 Archive（可删除）

### 领域卡片 = 骨架 = 目录

领域是你思想中的核心主题空间。所有概念、模型、冲突、实体都在领域内生长。

```
教学领域（顶级领域卡片）
├── 涉及概念：认知脚手架、最近发展区...
├── 涉及模型：学习螺旋模型...
├── 涉及方法：费曼技巧...
├── 涉及实体：Vygotsky、Bruner...
├── 涉及冲突：支持 vs 自主的张力
└── 子领域：课堂互动、课程设计...
```

AI 的工作方式：

1. 读取新内容
2. 扫描所有**顶级领域卡片**
3. 判断内容涉及哪些领域
4. 读取相关领域涉及的实例卡片
5. 进行比较、补充、冲突检测
6. 更新领域卡片和元结构

这就是"渐进式披露"——AI 不需要一次读取全部卡片，而是通过领域卡片树层层深入。

## 工作流

### 你做的事

```
1. 读完一篇文章 / 写完一个想法
2. 放入 00-Inbox/
3. 对 AI 说：
   - 「/construct」← 第一次或需要重建骨架时
   - 「/digest」   ← 日常消化新内容
   - 「/analyze 如何理解 X」← 让 AI 基于你的知识库回答问题
```

### AI 做的事

**`/construct`**：

```
读取种子内容 → 生成领域卡片 → 生成实例卡片 → 设计目录结构
→ 撰写 ontology.md → 输出建构报告 → 归档种子文档
```

**`/digest`**：

```
读取新内容 → 扫描领域卡片 → 读取相关实例卡片 → 比较与补充
→ 评估 Profile 变化 → 调整目录结构 → 更新领域卡片 → 归档原始文档
```

**`/analyze`**：

```
解析问题 → 读取 Profile → 读取领域与实例卡片 → 综合分析
→ 生成回答 → 输出到 04-OutBox/
```
