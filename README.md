# 新引擎 —— 关系驱动的知识Wiki

> **"知识不是被收集的，而是被消化的。"**
>
> 这是一套以**关系卡片**为骨架、以**冲突**为动力、以**你的认知档案**为透镜的关系驱动知识系统。

---

## 核心设计哲学

### 文章是燃料，卡片是引擎

外来文章和你自己的创作不是被"分解"成碎片的，而是被"消化"进已有卡片的燃料。

- 卡片数量增长缓慢
- 每张卡片内容不断丰富
- 原始文档消化后进入 Archive（可删除）

### 四种卡片类型

| 类型 | 功能 | 类比 |
|------|------|------|
| **概念** | 承载抽象教育/学习概念 | Wiki词条 |
| **实体** | 承载具体的人、机构、作品 | 人物传记/机构介绍 |
| **关系** | **核心创新**。既是逻辑关系的记录，也是知识的目录与导航枢纽 | 分类骨架+连线 |
| **冲突** | 记录矛盾与张力，不急于解决 | 辩证空间 |

### 关系卡片 = 中心卡片 = 目录卡片

这是本系统最核心的设计。

**关系卡片**不仅是"A和B有什么关系"的记录，它是整个知识网络的**骨架**：

```
教学关系（顶级关系卡片）
├── 支持关系（子关系卡片）
│   ├── 涉及概念：认知脚手架、最近发展区...
│   ├── 涉及实体：Vygotsky、Bruner...
│   ├── 涉及冲突：支持vs自主的张力
│   └── 子关系：脚手架关系、反馈关系...
├── 评价关系
├── 权力关系
└── 互动关系
```

**AI的工作方式**：
1. 读取新文章
2. 扫描所有**顶级关系卡片**（数量少，5-10张）
3. 判断文章涉及哪些关系类型
4. 读取相关**子关系卡片**
5. 读取子关系涉及的具体**概念/实体/冲突卡片**
6. 进行比较、补充、冲突检测
7. 更新关系卡片（添加新链接）

这就是"渐进式披露"——AI不需要一次读取全部卡片，而是通过关系卡片树层层深入。

## 工作流

### 你做的事

```
1. 读完一篇文章 / 写完一个想法
2. 放入 00-Inbox/ 对应文件夹
3. 对 AI 说："消化这篇文章"
```

### AI 做的事

```
1. 读取文章
2. 扫描 01-Cards/relations/ 下的全部关系卡片
3. 判断文章涉及哪些关系类型
4. 读取这些关系涉及的具体卡片
5. 进行比较：
   - 这个概念已有卡片？→ 丰富那张卡片（补充例子、引用、角度）
   - 这个概念没有卡片？→ 新建一张有内容的卡片
   - 有冲突？→ 在卡片中标记，并考虑新建冲突卡片
6. 更新关系卡片（添加新发现的链接）
7. 将原始文档移动到 03-Archive/
```

### 卡片之间的生长逻辑

```
新文章进入
    ↓
被关系卡片"识别"（属于哪种关系？）
    ↓
滋养已有概念/实体卡片
    ↓
可能发现冲突 → 新建/更新冲突卡片
    ↓
关系卡片更新（新增链接）
    ↓
原始文档归档
```

## 卡片的内容标准

### 概念卡片：拒绝空模板

**错误的做法**（信息稀薄的空卡片）：
```markdown
# 认知脚手架

> 一句话定义

## 核心内涵
（留空）

## 为什么重要
（留空）
```

**正确的做法**（初始就有丰富内容的卡片）：
```markdown
# 认知脚手架

> 教师为学习者提供的临时性支持结构，帮助其完成当前独立能力尚无法完成的任务。

## 核心内涵

### 两个不可或缺的特征
1. **目标是赋能而非代替**
   - 脚手架支撑的是最近发展区内的任务
   - 不是长期弥补能力，而是帮助学习者"够到"更高层次

2. **最终宿命是被拆除**
   - 每次成功的支持都应降低学习者对该支持的依赖
   - 如果长期使用后依赖性不减反增，就是失败的"辅助轮"

### 三层作用
- **策略层**：流程框架、清单、实时提问
- **方法论层**：学科话语体系、思维可视化工具
- **信念层**：统摄性隐喻、基本模型

## 为什么重要

（你的原创思考）

## 实例

（你的课堂实例）
```

### 关系卡片：既是关系记录，也是目录

```markdown
# 支持关系

> 教学中，更有能力的他者向学习者提供的、旨在促进学习发生的各类支持。

## 上层关系
- [[教学关系]] — 本关系属于教学关系的子类别

## 子关系
- **脚手架关系** — 临时性、结构性支持
- **反馈关系** — 信息性支持
- **同伴互助关系** — 平等主体间的支持

## 涉及的概念
- [[认知脚手架]] — 临时性支持结构
- [[最近发展区]] — 支持的作用范围
- [[元认知]] — 支持的目标：培养自我调节

## 涉及的实体
- [[Lev-Vygotsky]] — 最近发展区理论
- [[Jerome Bruner]] — 脚手架术语的首创者

## 涉及的冲突
- [[支持vs自主的张力]] — 支持过度导致依赖

## 相关关系
- **权力关系** — 支持暗含权力不对等
- **评价关系** — 支持与评价往往交织
```

### 冲突卡片：记录张力，不急于解决

```markdown
# 支持vs自主的张力

> 教学中，提供支持与培养自主之间存在着根本性的紧张关系。

## 张力描述
过度支持可能导致依赖，过早撤除支持可能导致挫败。

## 立场A：支持优先
- 代表：Vygotsky, 脚手架理论
- 论据：学习发生在最近发展区内

## 立场B：自主优先
- 代表：某些进步主义教育实践
- 论据：真正的学习只能来自主体的主动建构

## 你的判断
（留空，供你根据情境判断）
```

## 与旧系统的区别

| 维度 | 旧系统 | 新引擎 |
|------|--------|--------|
| 层级 | Inbox→Compile→Ontology→Permanent | Inbox→Cards |
| 卡片产生 | 每篇文章产生N张空卡片 | 文章滋养已有卡片，只新建真正需要的 |
| 联系建立 | 手动/AI预设链接 | 通过关系卡片自然生长 |
| 冲突处理 | 无 | 专门的冲突卡片记录张力 |
| 原始文档 | 永久保留在Inbox | 消化后进入Archive（可删） |
| AI角色 | 提取碎片 | 比较、补充、发现冲突 |
| 内容标准 | 空模板待填 | 初始就有丰富内容 |
